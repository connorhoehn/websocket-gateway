From src/services/ivs-chat-service.js:
```javascript
class IvsChatService {
  constructor(messageRouter, logger, config = {}) {
    this.ivsClient = new IvschatClient({ region: process.env.AWS_REGION || 'us-east-1' });
    this.roomArn = process.env.IVS_CHAT_ROOM_ARN;
    this.enabled = !!this.roomArn; // Feature flag
  }

  async handleSendMessage(clientId, { channel, message }) {
    const command = new SendMessageCommand({
      roomIdentifier: this.roomArn,
      content: message,
      attributes: { channel, clientId }
    });
    await this.ivsClient.send(command);
  }
}
```

```
Gap 1: IVS Chat CDK deployment not automated

The IVS Chat feature is fully implemented (service, Lambda handler, CDK infrastructure),
but the CDK stack deployment is not automated or documented. Users who want to enable
IVS Chat must manually:
1. Deploy IvsChatStack via `cdk deploy IvsChatStack`
2. Set IVS_CHAT_ROOM_ARN environment variable from stack output
3. Configure Lambda VPC access for Redis connectivity
```

Gap 2: No migration strategy for existing chat data

implies migration of existing data. Currently:

- No logic exists to migrate existing LRU cache data to IVS
- Feature flag approach means channels either use LRU OR IVS, not both
