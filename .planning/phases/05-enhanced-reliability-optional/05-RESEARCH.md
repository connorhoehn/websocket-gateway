# Phase 5: Enhanced Reliability (Optional) - Research

**Researched:** 2026-03-03
**Domain:** Fault tolerance, connection recovery, AWS IVS Chat integration
**Confidence:** MEDIUM-HIGH

## Summary

Phase 5 addresses enhanced reliability through three distinct features: graceful Redis degradation, WebSocket reconnection with session recovery, and optional AWS IVS Chat integration for persistent messaging with moderation.

The Redis degradation pattern is well-documented with clear Node.js implementation strategies from official Redis documentation. The existing cursor-service.js already implements cache-aside pattern (write to local, then Redis) which provides a foundation for degradation. WebSocket reconnection patterns are mature with established exponential backoff and state restoration techniques. AWS IVS Chat is a managed service that replaces in-memory chat with persistent storage and Lambda-based moderation, though it requires careful migration from the existing chat-service.js architecture.

The key technical challenge is maintaining connection affinity during reconnection when using Application Load Balancer sticky sessions, as session cookies may expire or clients may route to different tasks. The solution involves storing session state externally (Redis or DynamoDB) and using session tokens that survive reconnection.

**Primary recommendation:** Implement Redis degradation first (REL-04) as it's foundational and leverages existing cache-aside pattern. Then add connection recovery (REL-05) with external session storage. IVS Chat integration (IVS-01/02/03) is truly optional and represents architectural shift from ephemeral to persistent chat.

### Pattern 1: Graceful Redis Degradation
**What:** Services detect Redis unavailability and fall back to local cache without dropping connections
**When to use:** All Redis-dependent operations (cursor updates, chat history, pub/sub)
**Example:**
```javascript
// Source: https://redis.io/docs/latest/develop/clients/nodejs/error-handling/
class ServiceWithFallback {
  constructor(redisClient, logger) {
    this.redis = redisClient;
    this.localCache = new Map();
    this.redisAvailable = true;

    // Monitor Redis connection health
    this.redis.on('error', (err) => {
      logger.error('Redis error:', err);
      if (['ECONNREFUSED', 'ECONNRESET', 'ETIMEDOUT'].includes(err.code)) {
        this.redisAvailable = false;
        logger.warn('Redis unavailable, using local cache');
      }
    });

    this.redis.on('ready', () => {
      this.redisAvailable = true;
      logger.info('Redis connection restored');
    });
  }

  async getData(key) {
    // Try Redis first if available
    if (this.redisAvailable) {
      try {
        const val = await this.redis.get(key);
        if (val != null) {
          // Sync to local cache
          this.localCache.set(key, val);
          return val;
        }
      } catch (err) {
        if (['ECONNREFUSED','ECONNRESET','ETIMEDOUT','EAI_AGAIN'].includes(err.code)) {
          this.logger.warn('Redis unavailable, falling back to local cache');
          this.redisAvailable = false;
        } else {
          throw err;
        }
      }
    }

    // Fallback to local cache
    return this.localCache.get(key);
  }

  async setData(key, value, ttl) {
    // ALWAYS write to local cache (cache-aside pattern)
    this.localCache.set(key, value);

    // Then try Redis if available
    if (this.redisAvailable) {
      try {
        await this.redis.setEx(key, ttl, value);
      } catch (err) {
        if (['ECONNREFUSED','ECONNRESET','ETIMEDOUT','EAI_AGAIN'].includes(err.code)) {
          this.logger.warn('Redis unavailable, cached locally only');
          this.redisAvailable = false;
        } else {
          throw err;
        }
      }
    }
  }
}
```

### Pattern 2: Session Token Reconnection
**What:** Client receives session token on initial connection, uses it to reconnect and restore state
**When to use:** WebSocket reconnection after network interruption or server restart
**Example:**
```javascript
// Source: https://developers.ringcentral.com/guide/notifications/websockets/session-recovery
// Server-side session management
class SessionService {
  constructor(sessionStore, logger) {
    this.store = sessionStore; // Redis or DynamoDB
    this.logger = logger;
  }

  async createSession(clientId, userContext) {
    const sessionToken = crypto.randomUUID();
    const sessionData = {
      clientId,
      userContext,
      subscriptions: [], // Will be populated as client subscribes
      createdAt: Date.now(),
      expiresAt: Date.now() + (24 * 60 * 60 * 1000) // 24 hours
    };

    await this.store.set(`session:${sessionToken}`, JSON.stringify(sessionData), 86400);
    return sessionToken;
  }

  async restoreSession(sessionToken) {
    const sessionData = await this.store.get(`session:${sessionToken}`);
    if (!sessionData) {
      return null;
    }

    const session = JSON.parse(sessionData);
    if (session.expiresAt < Date.now()) {
      await this.store.del(`session:${sessionToken}`);
      return null;
    }

    return session;
  }

  async updateSubscriptions(sessionToken, subscriptions) {
    const sessionData = await this.store.get(`session:${sessionToken}`);
    if (!sessionData) return;

    const session = JSON.parse(sessionData);
    session.subscriptions = subscriptions;
    await this.store.set(`session:${sessionToken}`, JSON.stringify(session), 86400);
  }
}

// Client reconnection flow
// 1. Client connects with ?sessionToken=xyz if reconnecting
// 2. Server validates token and restores subscriptions
// 3. Client reuses same clientId to maintain presence state
```

### Pattern 3: IVS Chat Integration with Message Review Handler
**What:** Intercept chat messages via Lambda before delivery, forward approved messages to WebSocket clients
**When to use:** When persistent chat with moderation is required (opt-in feature)
**Example:**
```javascript
// Source: https://docs.aws.amazon.com/ivs/latest/ChatUserGuide/chat-message-review-handler.html
// Lambda message review handler
exports.handler = async (event) => {
  const { Content, MessageId, Sender, RoomArn, Attributes } = event;

  // Moderation logic (e.g., check for profanity, spam)
  const isProfane = await checkForProfanity(Content);

  if (isProfane) {
    return {
      ReviewResult: 'DENY',
      Content: Content,
      Attributes: {
        Reason: 'Message contains inappropriate content'
      }
    };
  }

  // Approved - publish to WebSocket clients via Redis pub/sub
  // This happens AFTER IVS delivers to chat room
  await publishToWebSocket(RoomArn, {
    type: 'chat',
    action: 'message',
    message: {
      id: MessageId,
      clientId: Sender.UserId,
      content: Content,
      timestamp: new Date().toISOString()
    }
  });

  return {
    ReviewResult: 'ALLOW',
    Content: Content,
    Attributes: Attributes
  };
};

// Backend service creates IVS Chat tokens
const { IvschatClient, CreateChatTokenCommand } = require('@aws-sdk/client-ivschat');
const ivsClient = new IvschatClient({ region: 'us-east-1' });

async function generateChatToken(userId, roomArn) {
  const command = new CreateChatTokenCommand({
    roomIdentifier: roomArn,
    userId: userId,
    capabilities: ['SEND_MESSAGE', 'DELETE_MESSAGE'],
    sessionDurationInMinutes: 60
  });

  return await ivsClient.send(command);
}
```

### Anti-Patterns to Avoid
- **Blocking on Redis reconnection:** Don't halt service operations waiting for Redis to reconnect; use local cache immediately
- **Stateful WebSocket servers without external session store:** Don't store session state only in memory; ALB may route reconnections to different tasks
- **Synchronous IVS message forwarding:** Don't wait for IVS Lambda response before acknowledging to client; use async pub/sub pattern
- **Mixing IVS and in-memory chat:** Don't run both simultaneously; choose one based on requirements (ephemeral vs persistent)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Redis reconnection logic | Custom retry mechanism with timers | redis client built-in reconnection + error event handlers | Redis client handles exponential backoff, connection pooling, command queuing during reconnect |
| WebSocket session recovery | Custom session tokens with encryption | crypto.randomUUID() + external store (Redis/DynamoDB) | UUIDs provide sufficient entropy; complexity is in state restoration, not token generation |
| Chat message moderation | Custom profanity filters or ML models | AWS IVS Chat message review handlers | Lambda-based handlers integrate with existing AWS ecosystem; IVS provides delivery guarantees |
| Message persistence | Custom database schema for chat history | AWS IVS Chat room history | IVS manages storage, pagination, TTL, and provides REST API for history retrieval |

**Key insight:** Distributed system reliability is about graceful degradation and state externalization, not custom retry logic. Use battle-tested client libraries (redis) and managed services (IVS Chat) that handle edge cases (split-brain scenarios, partial failures, clock drift).

## Common Pitfalls

### Pitfall 1: Redis Unavailability Causing Connection Drops
**What goes wrong:** When Redis connection fails, services throw errors and disconnect WebSocket clients even though local cache has valid data
**Why it happens:** Services don't register 'error' event listener on Redis client, causing uncaught exceptions; or they fail-fast on Redis errors instead of degrading gracefully
**How to avoid:**
1. ALWAYS register redis.on('error', handler) to prevent process crashes
2. Use connection state flags (redisAvailable) to route operations to local cache
3. Implement cache-aside pattern: write local first, then Redis
4. Monitor redis.on('ready') to restore Redis operations when connection recovers
**Warning signs:** Process crashes with "Unhandled 'error' event" from Redis client; connection count drops to zero when Redis restarts

### Pitfall 2: Reconnected Clients Route to Different Tasks (Lost State)
**What goes wrong:** Client reconnects with sessionToken but routes to different ECS task that doesn't have local session state, losing subscriptions and presence
**Why it happens:** ALB sticky sessions use cookies that may expire or be lost; tasks are ephemeral and local Maps/Sets don't survive task restarts; no external session store
**How to avoid:**
1. Store session state in Redis or DynamoDB, not in-memory Maps
2. Session token lookup restores subscriptions from external store
3. Presence service must re-register client on reconnection
4. Don't rely solely on ALB cookies; use application-level session tokens
**Warning signs:** Clients successfully reconnect but stop receiving messages; presence indicators disappear; "subscription not found" errors

### Pitfall 3: IVS Message Review Handler Becomes Single Point of Failure
**What goes wrong:** Lambda function errors or timeouts cause all messages to be denied/allowed based on FallbackResult, degrading user experience
**Why it happens:** Lambda cold starts, unhandled exceptions, or external API timeouts block message delivery; no circuit breaker or timeout handling
**How to avoid:**
1. Set FallbackResult to 'ALLOW' for user-facing errors (deny on explicit violations only)
2. Keep Lambda warm with CloudWatch Events (scheduled invocations)
3. Implement timeout in Lambda (process message in < 1 second)
4. Monitor Lambda errors/durations with CloudWatch alarms
5. Use simple moderation logic (keyword matching) before external API calls
**Warning signs:** Message delivery latency spikes; users report messages not sending; Lambda throttling errors in logs

### Pitfall 4: Session Token Replay Attacks
**What goes wrong:** Malicious client reuses old session token to impersonate another user or maintain unauthorized access after logout
**Why it happens:** Session tokens don't expire, aren't invalidated on logout, or lack tie to original client IP/user-agent
**How to avoid:**
1. Set session TTL (24 hours maximum, configurable based on security requirements)
2. Implement explicit logout that deletes session token from store
3. Validate userContext on every reconnection (ensure user still authorized)
4. Consider binding session to IP address or user-agent (though this may break mobile reconnections)
**Warning signs:** Security audit flags long-lived sessions; users report seeing messages from accounts they logged out of

### Pitfall 5: Local Cache Divergence During Redis Outage
**What goes wrong:** During Redis outage, each ECS task has different local cache state, causing inconsistent data across clients connected to different tasks
**Why it happens:** Local caches (Maps, LRU) are task-local; without Redis pub/sub, updates don't propagate across tasks
**How to avoid:**
1. Accept eventual consistency during outages (ephemeral data like cursors/presence)
2. Use Redis pub/sub status to determine if cross-task messaging is available
3. Show "degraded mode" indicator to users when Redis unavailable
4. Don't attempt to sync local caches without Redis (complexity > value)
5. For critical data (chat messages), consider buffering and replaying after Redis recovery
**Warning signs:** Users on different servers see different cursor positions; presence status inconsistent; chat messages lost during outage

### Exponential Backoff Reconnection
```javascript
// Source: https://redis.io/docs/latest/develop/clients/nodejs/error-handling/
async function getWithRetry(key, { attempts = 3, baseDelayMs = 100 } = {}) {
  let delay = baseDelayMs;
  for (let i = 0; i < attempts; i++) {
    try {
      return await redisClient.get(key);
    } catch (err) {
      if (
        i < attempts - 1 &&
        (['ETIMEDOUT','ECONNRESET','EAI_AGAIN'].includes(err.code) ||
         (err.name === 'ReplyError' && /(BUSY|TRYAGAIN|LOADING)/.test(err.message)))
      ) {
        await new Promise(r => setTimeout(r, delay));
        delay *= 2;
        continue;
      }
      throw err;
    }
  }
}
```

### WebSocket Reconnection with Session Recovery
```javascript
// Source: https://oneuptime.com/blog/post/2026-01-27-websocket-reconnection-logic/view
class ReconnectingWebSocket {
  constructor(url, options = {}) {
    this.url = url;
    this.reconnectDelay = options.reconnectDelay || 1000;
    this.maxReconnectDelay = options.maxReconnectDelay || 30000;
    this.reconnectAttempts = 0;
    this.sessionToken = options.sessionToken;
    this.connect();
  }

  connect() {
    const wsUrl = this.sessionToken
      ? `${this.url}?sessionToken=${this.sessionToken}`
      : this.url;

    this.ws = new WebSocket(wsUrl);

    this.ws.onopen = () => {
      this.reconnectAttempts = 0;
      this.reconnectDelay = 1000;
      console.log('Connected');
    };

    this.ws.onmessage = (event) => {
      const data = JSON.parse(event.data);

      // Save session token for reconnection
      if (data.type === 'session' && data.sessionToken) {
        this.sessionToken = data.sessionToken;
      }

      this.onmessage?.(data);
    };

    this.ws.onerror = (error) => {
      console.error('WebSocket error:', error);
    };

    this.ws.onclose = () => {
      console.log('Disconnected, reconnecting...');
      this.scheduleReconnect();
    };
  }

  scheduleReconnect() {
    this.reconnectAttempts++;

    // Add jitter to prevent thundering herd
    const jitter = Math.random() * 1000;
    const delay = Math.min(
      this.reconnectDelay * Math.pow(2, this.reconnectAttempts),
      this.maxReconnectDelay
    ) + jitter;

    setTimeout(() => this.connect(), delay);
  }
}
```

## Open Questions

1. **Session Token Storage Location**
   - What we know: Can use Redis (fast lookups) or DynamoDB (durable, on-demand pricing)
   - What's unclear: Performance impact of DynamoDB GetItem (5-20ms) on reconnection path vs Redis (1-2ms); cost at scale
   - Recommendation: Start with Redis for session tokens (fast, already deployed); fall back to DynamoDB if Redis unavailable (doubles as degradation test)

2. **IVS Chat Message Forwarding Latency**
   - What we know: Lambda message review handler is synchronous (blocks SendMessage); approved messages must be published to WebSocket clients via Redis pub/sub
   - What's unclear: End-to-end latency from client sends message → Lambda approves → pub/sub → WebSocket delivery; whether this meets <50ms target
   - Recommendation: Measure latency in Wave 0 test; if > 50ms, consider IVS opt-out for low-latency channels

3. **Handling Partial Redis Failures**
   - What we know: Redis client reconnects automatically; services can detect unavailability via connection events
   - What's unclear: Behavior during Redis failover in Multi-AZ ElastiCache (expected downtime 30-60 seconds); whether command queue overflows
   - Recommendation: Test failover scenario in staging; implement connection health endpoint to expose Redis status for monitoring
