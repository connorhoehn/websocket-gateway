## Opportunities surfaced (worth queuing)

### `cluster.scope('pipeline')` is documented but never called
`social-api/src/pipeline/bootstrap.ts:14` lists `cluster.scope('pipeline')` as a "feature the facade gives us for free." A grep of the entire codebase confirms it's referenced only in comments. The other consumer (`src/services/room-ownership-service.js`) does call `cluster.scope('rooms')` properly, so the feature works тАФ pipelines just aren't using it. If pipelines ever introduce locks/elections, scoping should be the first thing wired in.

### `ScopedLock` / `QuorumDistributedLock` have zero adoption surface today
ROI item #8 ("ScopedLock / QuorumDistributedLock") was tagged "drop-in if used at all today" тАФ confirmed no work. The pipeline doesn't take any locks; rooms takes `cluster.scope('rooms').lock` via the older `DistributedLock` API. **Dropped as a stream.** Bank for whenever pipelines need a critical-section.

### `PipelineModule` does not thread EventBus configuration from `PipelineModuleConfig`
**Streams 3, 4, and part of 5 are blocked on upstream distributed-core work.** Specifically: `src/applications/pipeline/PipelineModule.ts:271-274` constructs the EventBus inline with only `topic` and `walFilePath` тАФ it ignores `deadLetterHandler`, `metrics`, `autoCompactIntervalMs`, `autoCompactOptions` even though `EventBus`/`EventBusFactory.LocalEventBusConfig` accept them.

Required upstream change (тЙИ30 LOC in distributed-core):
- Add fields to `PipelineModuleConfig`: `eventBusDeadLetterHandler`, `eventBusMetrics`, `eventBusAutoCompactIntervalMs`, `eventBusAutoCompactOptions`
- Thread them into the `new EventBus<PipelineEventMap>(...)` call in `onInitialize()`

Until this lands: only the run-level metrics path (`PipelineModuleConfig.metrics` тЖТ run started/completed/failed/cancelled counters) is reachable; the bus itself is an unobserved black box.

### `RaftRpcSigner` lives on `RaftConfig.signer`, not `ClusterConfig`
Stream 8 (RpcSigning) only takes effect when `clusterConfig.registry.type === 'raft'` тАФ the signer is a field on `RaftConfig`, threaded through `RegistryConfig`'s raft branch. So Stream 8 is **dependent on Stream 7** (raft mode enabled), not an independent concern. Single-process deployments using `'memory'` or `'wal'` registry never see signed RPCs because there are no inter-node RPCs to sign.

### `FailureDetectorBridge`, `PubSubHeartbeatSource`, `HintedHandoffQueue` are standalone classes тАФ not part of `Cluster.create()`
Streams 9 and 10 don't modify `ClusterConfig`. They construct on top of an already-started cluster:
```ts
const bridge = new FailureDetectorBridge(cluster.failureDetector, { ... });
const hhq = new HintedHandoffQueue({ ... });
```
This means the modularization seam for those streams is a `setup*(cluster, opts)` helper called after `cluster.start()`, not a config builder.

### `IRaftStateMachine` is generic тАФ pipeline runs map cleanly
`distributed-core/src/cluster/raft/state/IRaftStateMachine.ts` exposes `<TCommand, TResult>` plus `apply` / `applyBatch` / `exportSnapshot` / `importSnapshot` and `lastAppliedIndex` / `lastAppliedTerm` / `clusterConfig`. The base class `RaftStateMachineBase` handles `'config-change'` and `'noop'` automatically. Stream 6 only has to define the pipeline-run command shape and the `apply` semantics.

### `PubSubConfig` name collision in distributed-core
The top-level `distributed-core` barrel re-exports `PubSubConfig` from `gateway/pubsub/types` (the runtime config interface тАФ `enableCrossNodeDelivery`, `messageDeduplicationTTL`). But `ClusterConfig.pubsub` is typed against a different `PubSubConfig` declared in `cluster/Cluster.ts` (the discriminated union тАФ `{ type: 'memory' | 'redis' }`). Importing `PubSubConfig` from `distributed-core` and using it as the return type of a builder produces a type error because the wrong shape wins.

Workaround in `social-api/src/pipeline/config/cluster.ts`: derive the type via `ClusterConfig['pubsub']` indexed access. Real fix is upstream тАФ rename one of the two so the barrel surface is unambiguous (e.g., the cluster-side becomes `ClusterPubSubConfig` since it's already wrapped as `ClusterConfig['pubsub']`).

## Additional adoption candidates surfaced from a HEAD survey

Beyond the original 10-item ROI table. Each is **not in flight** in the current streams; recording so they can be triaged for a Wave 3 or backlog. Roughly ranked by relevance to a single-process social-api today vs. multi-node tomorrow.

### Worth queuing for **Wave 3 (single-process today)**

1. **`AutoCheckpointer` + `RecoveryManager`** (persistence) тАФ currently the gateway sets `PIPELINE_WAL_PATH` but has no checkpoint strategy. WAL grows unbounded. AutoCheckpointer triggers on LSN threshold or time interval. Distinct from EventBus auto-compaction (which Stream 4 will cover) тАФ this is the *entity registry / state store* WAL. Effort: ~half day. Cleanest on top of an already-modularized bootstrap.

2. **`testing/` subpath helpers** тАФ there's a `'distributed-core/testing'` subpath export with fixtures and helpers. The gateway already uses `FixtureLLMClient` from the main barrel. Investigate whether the testing subpath has cluster/test-harness primitives that would simplify `ownership-events.test.ts` or future Raft integration tests. Effort: ~1 hour of investigation, then unknown if anything's adopted.

3. **`UnifiedPersistenceFactory`** (persistence) тАФ single factory for all persistence backends. Today the gateway threads `walPath` through `RegistryConfig` and `LocalEventBusConfig` separately. UnifiedPersistenceFactory may consolidate this. Effort: ~half day if it's a clean refactor.

4. **`ChaosInjector` + `DiagnosticTool`** (diagnostics) тАФ wire into a test-only suite that validates pipeline resilience under failure conditions. Useful for staging / pre-prod environments. Effort: ~1 day, but pure addition (no risk to existing behavior).

5. **`ObservabilityManager`** (cluster/observability) тАФ `ClusterDashboard`, `TopologyQuery`, `ClusterScalingAnalysis`, `ResourcePlacementRecommendation`. The gateway already has `/internal/postmortem` (uses `cluster.snapshot()`); `ObservabilityManager` is the structured/queryable variant. Could feed the frontend's observability dashboard. Effort: ~1-2 days incl. wiring an HTTP route.

### Worth queuing for **Wave 4 (multi-node prep)**

6. **`RebalanceManager` + `RebalancePolicy`** (routing) тАФ auto-rebalance pipeline-run ownership when topology changes. Today, when a node leaves, runs are reassigned via `pipeline.run.reassigned` from `PipelineModule`'s member-left handler. RebalanceManager is the lower-level primitive тАФ could replace the bridge-level reassignment with a more proactive policy. Multi-node only.

7. **`ClusterLeaderElection` / `DistributedSession`** тАФ leader election (multi-node primary picking) and cross-node session state. The gateway has its own session reconnection story today; DistributedSession would let sessions survive node loss. Multi-node only; non-trivial integration.

8. **`MessageBatcher.wrapTransport` + `EncryptedTransport`** тАФ transport-level adoptions. MessageBatcher for throughput, EncryptedTransport for cross-AZ TLS-like encryption. Both wrap an existing transport. Multi-node only.

9. **Zone-aware components** (`ZoneTopology`, `ZoneQuorumPolicy`, `RegionSelector`, `GeographicGossip`, `ZoneReplicationPolicy`) тАФ multi-AZ correctness primitives that landed in v0.8.0-rc P-phases. Defer until you commit to a multi-AZ deploy plan; that's a serious migration of its own.

10. **`PeerMessaging` / `PeerEnvelope`** тАФ higher-level p2p messaging that replaces some lower-level transport sends. Multi-node only.

### Worth a one-time evaluation, then probably **drop**

- **`createCluster` / `NodeHandle` / front-door API** тАФ alternative to `Cluster.create()`. Gateway already uses the facade; the front door's main advantage is multi-node bootstrap, which the facade also handles. Probably no migration value.
- **`cli` entrypoint** тАФ standalone node CLI. Gateway's lifecycle is HTTP-server-driven, not CLI-driven. Drop.
- **`agent` framework** тАФ for management agent sidecars. No current use case here.
- **`benchmarks/` subpath** тАФ distributed-core's own benchmarks; gateway doesn't need to integrate.

## Risks observed

### `bootstrap.ts` runs distinct fastTimers / registry paths
Two divergent code paths inside `bootstrap.ts`:
1. **Test path (`fastTimers === true`)**: bypasses `moduleRegistry.register()` and calls `module.initialize() + module.start()` directly with a quiet logger context. Disables registry-side lifecycle tracking.
2. **Production path**: `moduleRegistry.register(module, { configuration: { pubsub, metrics } })`.

Both paths must behave identically post-refactor; any stream that touches `pipelineModule` config has to keep both working. The fastTimers branch in particular has hand-rolled `quietContext` construction that any new `PipelineModuleConfig` field would also need to be threaded into.

### Pre-existing identity-collision behavior in tests
`bootstrap.ts:262-268` documents that test mode deliberately mints ephemeral identities even when `PIPELINE_IDENTITY_FILE` is set, because "two sequential bootstraps in the same jest run produces id collisions and breaks the existing 'distinct nodeIds' suite." Stream 7 (Raft) needs to make sure the Raft cluster identity doesn't collide either тАФ a single Raft group can't have two members with the same nodeId. Probably not an issue since Raft tests would set up multiple processes anyway, but worth flagging.
