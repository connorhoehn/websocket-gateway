## Opportunities surfaced (worth queuing)

### `cluster.scope('pipeline')` is documented but never called
`social-api/src/pipeline/bootstrap.ts:14` lists `cluster.scope('pipeline')` as a "feature the facade gives us for free." A grep of the entire codebase confirms it's referenced only in comments. The other consumer (`src/services/room-ownership-service.js`) does call `cluster.scope('rooms')` properly, so the feature works — pipelines just aren't using it. If pipelines ever introduce locks/elections, scoping should be the first thing wired in.

### `ScopedLock` / `QuorumDistributedLock` have zero adoption surface today
ROI item #8 ("ScopedLock / QuorumDistributedLock") was tagged "drop-in if used at all today" — confirmed no work. The pipeline doesn't take any locks; rooms takes `cluster.scope('rooms').lock` via the older `DistributedLock` API. **Dropped as a stream.** Bank for whenever pipelines need a critical-section.

### `PipelineModule` does not thread EventBus configuration from `PipelineModuleConfig`
**Streams 3, 4, and part of 5 are blocked on upstream distributed-core work.** Specifically: `src/applications/pipeline/PipelineModule.ts:271-274` constructs the EventBus inline with only `topic` and `walFilePath` — it ignores `deadLetterHandler`, `metrics`, `autoCompactIntervalMs`, `autoCompactOptions` even though `EventBus`/`EventBusFactory.LocalEventBusConfig` accept them.

Required upstream change (≈30 LOC in distributed-core):
- Add fields to `PipelineModuleConfig`: `eventBusDeadLetterHandler`, `eventBusMetrics`, `eventBusAutoCompactIntervalMs`, `eventBusAutoCompactOptions`
- Thread them into the `new EventBus<PipelineEventMap>(...)` call in `onInitialize()`

Until this lands: only the run-level metrics path (`PipelineModuleConfig.metrics` → run started/completed/failed/cancelled counters) is reachable; the bus itself is an unobserved black box.

### `RaftRpcSigner` lives on `RaftConfig.signer`, not `ClusterConfig`
Stream 8 (RpcSigning) only takes effect when `clusterConfig.registry.type === 'raft'` — the signer is a field on `RaftConfig`, threaded through `RegistryConfig`'s raft branch. So Stream 8 is **dependent on Stream 7** (raft mode enabled), not an independent concern. Single-process deployments using `'memory'` or `'wal'` registry never see signed RPCs because there are no inter-node RPCs to sign.

### `FailureDetectorBridge`, `PubSubHeartbeatSource`, `HintedHandoffQueue` are standalone classes — not part of `Cluster.create()`
Streams 9 and 10 don't modify `ClusterConfig`. They construct on top of an already-started cluster:
```ts
const bridge = new FailureDetectorBridge(cluster.failureDetector, { ... });
const hhq = new HintedHandoffQueue({ ... });
```
This means the modularization seam for those streams is a `setup*(cluster, opts)` helper called after `cluster.start()`, not a config builder.

### `IRaftStateMachine` is generic — pipeline runs map cleanly
`distributed-core/src/cluster/raft/state/IRaftStateMachine.ts` exposes `<TCommand, TResult>` plus `apply` / `applyBatch` / `exportSnapshot` / `importSnapshot` and `lastAppliedIndex` / `lastAppliedTerm` / `clusterConfig`. The base class `RaftStateMachineBase` handles `'config-change'` and `'noop'` automatically. Stream 6 only has to define the pipeline-run command shape and the `apply` semantics.

### `PubSubConfig` name collision in distributed-core
The top-level `distributed-core` barrel re-exports `PubSubConfig` from `gateway/pubsub/types` (the runtime config interface — `enableCrossNodeDelivery`, `messageDeduplicationTTL`). But `ClusterConfig.pubsub` is typed against a different `PubSubConfig` declared in `cluster/Cluster.ts` (the discriminated union — `{ type: 'memory' | 'redis' }`). Importing `PubSubConfig` from `distributed-core` and using it as the return type of a builder produces a type error because the wrong shape wins.

Workaround in `social-api/src/pipeline/config/cluster.ts`: derive the type via `ClusterConfig['pubsub']` indexed access. Real fix is upstream — rename one of the two so the barrel surface is unambiguous (e.g., the cluster-side becomes `ClusterPubSubConfig` since it's already wrapped as `ClusterConfig['pubsub']`).

## Risks observed

### `bootstrap.ts` runs distinct fastTimers / registry paths
Two divergent code paths inside `bootstrap.ts`:
1. **Test path (`fastTimers === true`)**: bypasses `moduleRegistry.register()` and calls `module.initialize() + module.start()` directly with a quiet logger context. Disables registry-side lifecycle tracking.
2. **Production path**: `moduleRegistry.register(module, { configuration: { pubsub, metrics } })`.

Both paths must behave identically post-refactor; any stream that touches `pipelineModule` config has to keep both working. The fastTimers branch in particular has hand-rolled `quietContext` construction that any new `PipelineModuleConfig` field would also need to be threaded into.

### Pre-existing identity-collision behavior in tests
`bootstrap.ts:262-268` documents that test mode deliberately mints ephemeral identities even when `PIPELINE_IDENTITY_FILE` is set, because "two sequential bootstraps in the same jest run produces id collisions and breaks the existing 'distinct nodeIds' suite." Stream 7 (Raft) needs to make sure the Raft cluster identity doesn't collide either — a single Raft group can't have two members with the same nodeId. Probably not an issue since Raft tests would set up multiple processes anyway, but worth flagging.
